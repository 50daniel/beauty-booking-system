# AGENT.md

## 專案說明讀取規則

- 開始處理任何專案相關任務前，必須先讀取 `README.md` 作為專案說明。
- 若 `README.md` 有更新，應以最新內容為準。
- 回答或修改程式碼時，需遵循專案說明中的架構、啟動方式、測試方式與限制。

## 角色與協作方式

使用者是一位軟體工程師。進行開發任務時，請以工程品質、可維護性與清楚溝通為優先。

在開始寫程式之前，若任務不是明確的小修改，請先提出：

- 預計採用的架構或實作方向
- 可能影響的檔案或模組
- 重要取捨與風險
- 預計驗證方式

待使用者確認後，再開始執行。

若任務描述、需求邊界、輸入輸出格式、環境限制或預期行為不明確，請先詢問使用者確認。不要自行假設後大幅修改，也不要做與任務無關的重構。

## 程式風格

寫程式時請遵守以下原則：

- 優先使用簡單、直接、容易理解的實作。
- 遵循既有專案的架構、命名、格式與慣例。
- 保持模組職責清楚，避免不必要的抽象。
- 避免過度工程化，只有在能實際降低複雜度或重複時才新增抽象。
- 讓函式與變數命名清楚表達意圖。
- 控制函式大小與巢狀層級，讓主要流程容易閱讀。
- 不為了通過測試而寫特殊案例硬編碼。
- 不任意改動與任務無關的程式碼、格式或設定。

## 套件與 API 使用

使用套件或 API 時：

- 優先使用專案既有依賴與標準函式庫。
- 新增套件前，請說明為什麼需要它，以及是否有更簡單的替代方案。
- 對於比較不常見、語意不直覺或容易誤用的套件函式，請在回覆中提供精簡說明，包含：
  - 該函式的用途
  - 本次使用方式
  - 重要參數或回傳值
- 不要加入大型依賴來解決小問題，除非理由充分。

## 實作流程

建議流程：

1. 先閱讀相關程式碼與設定，理解現有設計。
2. 若需求需要設計判斷，先提出方案並等待確認。
3. 實作時維持修改範圍精準。
4. 補上必要測試或調整既有測試。
5. 執行合理的驗證指令。
6. 回覆時說明完成內容、驗證結果，以及任何尚未處理的限制。

## 溝通規則

- 回覆請清楚、精簡、直接。
- 若有不確定處，明確指出並詢問。
- 若發現需求可能造成風險或設計問題，請先提出原因與替代方案。
- 若無法執行測試或驗證，請明確告知原因。
- 不要假裝已完成未實際完成的工作。

## 禁止事項

- 不要在需求不明確時自行大幅擴充功能。
- 不要任意重構無關程式碼。
- 不要覆蓋或還原使用者既有修改，除非使用者明確要求。
- 不要加入未說明用途的新依賴。
- 不要留下難以理解的複雜實作。
